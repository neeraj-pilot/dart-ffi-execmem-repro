Why a Flutter AOT app can trip GrapheneOS's DCL protection

A native-to-Dart callback can require a small executable trampoline even when the application itself was compiled ahead of time.

Verified 10 July 2026 · Android ARM64 · Flutter 3.38.10 · Dart 3.10.9

View repository Run the reproduction

The short version

Flutter release applications are AOT compiled, but AOT does not remove every runtime adapter. When native code needs to call a Dart function, the Dart VM exposes a native function pointer backed by a small callback trampoline. On Android, the VM copies that trampoline into writable memory and then asks the kernel to make the page executable.

GrapheneOS can block this transition through its per-app Dynamic code loading via memory restriction. The kernel returns EACCES; the current Dart VM treats the failed mprotect as fatal and aborts the process.

The missing distinction

Calling native code is not the same as being called by it

Ordinary Dart-to-native calls jump to machine code already present in a loaded library. A native-to-Dart callback has a different problem: the native caller only knows how to invoke a machine-code address, while the destination is managed by the Dart VM.

Where the failure occurs

  1. The Dart VM allocates a non-executable trampoline page.
  2. It copies the prebuilt trampoline instructions into that page.
  3. It requests read/execute protection with mprotect.
  4. GrapheneOS's restricted policy returns EACCES.
  5. The VM reports a fatal error and sends SIGABRT.
This is not application JIT compilation. The Dart callback body remains AOT compiled. The runtime operation creates an adapter that lets native code enter the VM correctly.

Case study

How Cronet exposed the behavior in Ente Auth

Ente issue #5723 reports that Auth stopped launching on GrapheneOS when DCL via memory was restricted. The last working release preceded Ente's adoption of the Android native Dio adapter, which routes requests through Cronet and first shipped in Auth 4.3.5.

Cronet reports headers, data, completion, and failures through Java callbacks. Its generated JNI proxy exposes a Dart callback as a native function pointer using Pointer.fromFunction. That reaches the same VM path as the dependency-free reproduction.

Cronet 1.9.0 is named only because it is the version used by the controlled test. This investigation is not rebutting a claim from Cronet maintainers about executable-memory restrictions.

Field evidence

Ente Auth 4.3.5 on GrapheneOS produced the same mprotect(EACCES) abort when the per-app restriction was enabled.

Controlled evidence

Independently denying only executable mprotect requests reproduces the abort, while the no-callback app remains alive.

Source correlation

Current Dart VM source performs the RW-to-RX transition, and the tested Cronet binding still constructs the callback function pointer through Dart FFI.

This pattern is broader than Cronet

The trigger is the callback API, not networking. The media_kit project documented the same Dart VM behavior and added an isolate-based fallback. Drift's native SQLite backend registers Dart SQL functions, and the SQLite binding exposes those functions through NativeCallable. That makes it a candidate for the same mechanism, but this repository does not test SQLite. None of this means every native Flutter package is affected: packages using platform channels, precompiled bridges, or only Dart-to-native FFI calls may avoid this path entirely.

Engineering options

The fix can live at three layers

01

Application workaround

Select a backend that avoids the callback bridge. For Ente Auth, that could mean Dio's dart:io adapter on Android instead of the native Cronet adapter.

02

Package-level bridge

Use a precompiled JNI/native bridge, platform channels, native ports, or polling where the native API permits it. This is contained but package-specific.

03

Dart VM strategy

Provide Android AOT callback stubs or dispatch that does not require a runtime writable-to-executable transition. This is the systemic and most complex option.

Allowing DCL via memory is a user compatibility setting, not an engineering fix. It disables a hardening policy the user deliberately selected.

For Flutter engineers

A practical diagnostic checklist

  1. Reproduce with a release build so debug JIT does not confuse the signal.
  2. Look for mprotect failed: 13 in the native tombstone.
  3. Bisect the dependency or backend change that introduced the crash.
  4. Search source and generated bindings for Pointer.fromFunction and NativeCallable.
  5. Run a no-callback negative control under the same restriction.
  6. Do not blame Flutter generally just because the fatal check lives in libflutter.so.
